Investigation Tips

The first point to start in the investigation is the bugReportTemplate.txt . This is found in the downloaded artifacts.

Lynn H has an analysis script that can point to some hints to possible failure points.

# !bin/csh -f
find . -name hs_err\*.log -print
find . -name jrockit.\*.dump -print
find . -name core -print
find . -name \*.hprof -print

fgrep OutOfMemory *.log */*.log | grep -v HeapDumpOnOutOfMemoryError
fgrep OutOfSharedMemory *.log */*.log
fgrep NullPointerException *.log */*.log
fgrep ConcurrentModificationException *.log */*.log
fgrep Assert *.log */*.log
fgrep deadlock *.log */*.log
fgrep InternalGemFire *.log */*.log
fgrep "Too many" *.log */*.log
fgrep "Suspect notification" *.log */*.log
fgrep "NoSuchMethod" *.log */*.log
fgrep "NoClassDef" *.log */*.log

grep Exception *.log */*.log | grep -v continuing | grep -v HeapDumpOnOutOfMemoryError

Try to figure out the life-cycle of the Object, like where was it created, trace it propagation through the distributed system. grep is your friend

The following example commands may help figure out where everything is. Replace the object ids and machine tags.

grep -r "Object_9287: after" .
grep -r "Listener.*Object_1998" .
